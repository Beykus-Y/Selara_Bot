# Известные баги и расхождения механик (выявлено по коду)

> Ниже не «гипотезы», а проблемы/расхождения, которые видны при анализе конкретных участков кода и напрямую влияют на поведение для пользователя.

---

## 1) `text_commands_locale=en` фактически отключает встроенные текстовые команды

### Как проявляется для пользователя

- В группе включены текстовые команды,
- но при `text_commands_locale=en` фразы вроде `who am i`/`help` (и вообще built-in text) не работают.
- При этом slash-команды продолжают работать.

### Причина в коде

В обработчике текстовых команд есть ранний выход при `locale != ru`: встроенные текстовые команды не обрабатываются, остаются только smart-triggers/custom RP. Это делает `en` скорее режимом «без встроенных текстовых команд», а не полноценной английской локалью.

### Где смотреть

- `src/selara/presentation/handlers/text_commands.py` (ветка `if chat_settings.text_commands_locale.lower() != "ru": ... return`).

### Влияние

Пользователь ожидает английские текстовые команды, но получает «тишину».

---

## 2) Текстовая команда `роль <game_id>` не поддерживается, хотя slash `/role <game_id>` поддерживается

### Как проявляется

- `/role game_xxx` в ЛС может показать роль конкретной игры.
- `роль game_xxx` не распознаётся как текстовая команда.

### Причина в коде

Валидация хвоста для текстового `role` жёстко возвращает `False`, поэтому команда с аргументом не проходит матчинг.

### Где смотреть

- `src/selara/presentation/commands/catalog.py` (`if command_key == "role": return False`).
- `src/selara/presentation/handlers/game/router.py` (`/role` принимает args и умеет открывать конкретную игру).

### Влияние

Непоследовательность UX: slash и текст ведут себя по-разному на одинаковый сценарий.

---

## 3) Текстовое `когда был @username` не резолвится как таргет-команда

### Как проявляется

- Пользователь пишет `когда был @username` и ожидает last seen по конкретному человеку.
- Текстовый резолвер не разбирает это как таргетную форму.

### Причина в коде

`когда был` и `когда была` заданы как **exact aliases** без парсинга хвоста. Если добавить аргумент, exact-match не срабатывает, а отдельного prefix-резолвера для этой команды нет.

### Где смотреть

- `src/selara/presentation/commands/aliases.py` (exact-триггеры `когда был`, `когда была`).
- `src/selara/presentation/commands/resolver.py` (exact-match требует полное совпадение строки).

### Влияние

Пользователю приходится переходить на `/lastseen @username` или reply-вариант.

---

## 4) В `/pay` reply-цель имеет приоритет над текстовым `@username`, что может неожиданно отправить монеты не туда

### Как проявляется

Сценарий:
1. Пользователь делает reply на сообщение A.
2. Пишет что-то вроде `/pay @B 100`.
3. Перевод идёт reply-цели A (если сумма распознана), а не `@B`.

### Причина в коде

Если есть `reply_to_message`, получатель берётся из reply. Текстовые токены в этом режиме используются только для суммы (и то через последний токен-число).

### Где смотреть

- `src/selara/presentation/handlers/economy.py`, `pay_command`.

### Влияние

Риск ошибочного перевода по невнимательности пользователя.

---

## 5) `/pay` в reply-режиме даёт неинформативную ошибку при неправильной сумме

### Как проявляется

- Reply + `/pay сто` (или любой нечисловой хвост).
- Вместо точной подсказки по формату приходит общий текст: «Не удалось определить получателя/сумму».

### Причина в коде

В reply-ветке сумма устанавливается только если последний токен — число; иначе в конце срабатывает общий fallback.

### Где смотреть

- `src/selara/presentation/handlers/economy.py`, `pay_command`.

### Влияние

Плохая диагностируемость для конечного пользователя.

---

## 6) Предложения пары/брака допускают ввод `user_id`, даже если пользователь фактически не в чате

### Как проявляется

- Пользователь может указать числовой `user_id`, для которого нет нормального присутствия в текущем чате.
- Предложение создаётся, но практический UX может быть «висячим» до истечения срока.

### Причина в коде

Резолвер цели для отношений при numeric id может создать snapshot даже без полноценного chat-контекста участника.

### Где смотреть

- `src/selara/presentation/handlers/relationships.py`, `_resolve_target_user`.
- `src/selara/presentation/handlers/relationships.py`, `_send_relationship_proposal`.

### Влияние

Пользователь считает, что «бот не довёл предложение», хотя проблема в таргете вне реального контекста.

---

## 7) Несимметричный UX: часть команд поддерживает reply+@username, часть — только reply

### Как проявляется

Пользователь ожидает единый стиль ввода, но получает разное поведение:
- `pair/marry/adopt/pet/family` умеют reply и `@username/user_id`.
- Социальные действия (`обнять`, `поцеловать`, и т.д.) — строго reply-only.

### Причина в коде

Социальные действия проверяют только `message.reply_to_message`, без попытки разобрать `@username`.

### Где смотреть

- `src/selara/presentation/handlers/text_commands.py`, `_send_social_action`.

### Влияние

Пользовательская путаница «почему тут @username работает, а тут нет».

## 8) `/start` в ЛС не равен «выдать роль сейчас»

### Как проявляется

Пользователь:
1. зашёл в игру с ролями,
2. открыл ЛС и нажал `/start`,
3. ожидает мгновенную выдачу роли,
4. получает только старт-панель/приветствие.

### Причина в коде

`/start` открывает приватную панель. Роль показывается отдельной логикой (`/role`, deep-link `game_...`, активная сессия и статус игры).

### Где смотреть

- `src/selara/presentation/handlers/game/router.py`, `start_command`.
- `src/selara/presentation/handlers/game/router.py`, `role_command`, `_show_role_for_user`.

### Влияние

Частое ложное ощущение «бот не выдал роль», хотя ожидание и реальная логика различаются.

## 9) Для `text_commands_locale=en` custom RP/trigger остаются, а built-in text исчезают — визуально выглядит как «частичный развал»

### Как проявляется

- Пользователь видит, что некоторые текстовые фразы всё ещё срабатывают (кастомные),
- но базовые встроенные команды — нет.

### Причина

Разные ветки в обработчике: custom trigger/rp проверяются до/вне built-in parser для `locale != ru`.

### Где смотреть

- `src/selara/presentation/handlers/text_commands.py` (ветки для `custom_rp_enabled`, `smart_triggers_enabled`, `locale != ru`).

### Влияние

Сложная для пользователя диагностика: «бот иногда понимает текст, иногда нет».

---

## 10) Источник пользовательского недоверия: одинаковая по смыслу команда может требовать разный формат ввода в разных модулях

### Наблюдение

Это не один crash-баг, а системная UX-проблема:
- где-то нужен reply,
- где-то можно `@username`,
- где-то только slash-форма надёжна,
- где-то текстовая форма неполная.

### Подтверждение по коду

Разные обработчики реализуют таргетинг независимо (`relationships`, `chat_assistant`, `economy`, `stats`, `text_commands`), без общего унифицированного адаптера «reply/@username/user_id».

### Влияние

Пользователь воспринимает это как «рандомную нестабильность», даже когда бот формально работает по правилам конкретного модуля.

---

## Что делать пользователю прямо сейчас

1. Для критичных операций используйте slash-формат.
2. Для адресных действий сначала пробуйте reply.
3. Для `lastseen` таргета используйте `/lastseen @username` вместо текста.
4. Для роли в игре используйте `/role` в ЛС.
5. Для переводов не смешивайте reply и отдельный `@username` в одной команде.
