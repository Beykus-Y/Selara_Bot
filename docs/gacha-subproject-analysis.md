# Gacha Analysis

## Почему отдельный подпроект

По структуре Selara это один async-монолит с тесной связью Telegram, web и общей инфраструктуры в `src/selara/`.
Для новой гачи безопаснее не трогать основной рантайм и не встраиваться в существующие use case прямо сейчас.

Отдельный сервис лучше подходит для вашего сценария, потому что:

- его можно поднять на другом VPS;
- он не требует немедленных изменений в командах бота;
- у него может быть отдельная база и собственный темп разработки;
- интеграцию с Selara можно отложить до готовности команды `гача генш`.

## Точка интеграции

Рекомендуемый первый способ:

1. Selara получает текстовую команду `гача генш`.
2. Telegram-хендлер вызывает `POST /v1/gacha/pull`.
3. Сервис возвращает:
   - готовый текст ответа;
   - данные карточки;
   - ссылку на изображение;
   - новое состояние игрока.
4. Selara отправляет текст и фото в Telegram.

Это предпочтительнее прямой работы через общую БД на первом этапе, потому что:

- не нужно связывать схемы двух сервисов;
- меньше риск сломать текущую БД Selara;
- проще разнести по VPS;
- легче контролировать контракт и ошибки.

## Что уже сделано в MVP

- создан отдельный подпроект `gacha/`;
- выбран FastAPI как совместимый с текущим стеком Selara;
- добавлено локальное async SQLite-хранилище;
- реализована базовая крутка по баннеру `genshin`;
- добавлены очки, примогемы, опыт и ранг приключений;
- добавлен кулдаун на 3 часа;
- сервис отдает готовое сообщение для Telegram.
